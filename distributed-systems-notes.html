Week 01Foundations

Introduction & System Models

1.1 What is a Distributed System?

Tanenbaum & Van Steen (2017)

A distributed system is a collection of autonomous computing elements that appears to its users as a single coherent system.

The two key properties in this definition are deliberately in tension:

  • Autonomous computing elements — nodes (processes, machines) operate independently and may fail independently.
  • Single coherent system — users should be unaware of the underlying distribution. This is the goal of distribution transparency.

1.2 Why Build Distributed Systems?

MotivationExplanation
Resource sharingShare hardware, data, and services across organisations
ScalabilityScale horizontally by adding more nodes, not just faster ones
Fault toleranceReplicate data and services so failures don't cause downtime
Geographical distributionPlace computation near users; comply with data sovereignty laws
ParallelismDivide large computational problems across many processors

1.3 Distribution Transparency

The ISO Reference Model for Open Distributed Processing defines eight forms of distribution transparency:

TransparencyHides…
AccessHow resources are accessed (local vs remote)
LocationPhysical or network location of a resource
MigrationThat a resource may move between nodes
RelocationThat a resource may move while in use
ReplicationThat multiple copies of a resource exist
ConcurrencyThat multiple users share a resource
FailureFailure and recovery of a resource
PersistenceWhether a resource is in memory or disk
Fundamental Tension

Full distribution transparency is unachievable. The network always introduces latency and the possibility of partial failure. Pretending otherwise leads to incorrect systems. Good distributed system design acknowledges and explicitly handles these failures rather than hiding them.

1.4 System Models

Before designing or analysing a distributed system, we must define a system model — a precise set of assumptions about what can and cannot fail. Three categories of model:

Physical Model

Describes the hardware — the set of nodes and communication links, generations of distributed hardware (early clusters, internet-scale systems, cloud-native).

Architectural Model

Describes how components are structured and interact:

  • Client-server — asymmetric roles; server provides service, client consumes.
  • Peer-to-peer (P2P) — all nodes are equivalent; roles are dynamic.
  • Layered — each layer provides services to the layer above, uses services from below.
  • Event-based (publish-subscribe) — producers publish events; consumers subscribe.

Fundamental Model

Makes precise assumptions about three aspects:

DimensionSynchronous ModelAsynchronous Model
Process timingEach step takes between [min, max] timeNo bound on step execution time
Message deliveryMessages delivered within known boundMessages may be delayed arbitrarily
Clock driftClock drift bounded by known constantClock drift is unbounded
Algorithms possibleEasier — timeouts work reliablyHarder — FLP impossibility applies
RealismRarely achievable in practiceAccurately models the internet

1.5 Failure Models

A precise taxonomy of how components can fail:

Failure TypeDescriptionSeverity
Crash failureProcess halts and stops sending messages. It was correct until it stopped.Low
Omission failureProcess fails to respond to some (not all) requests. Includes send omission and receive omission.Medium
Timing failureResponse falls outside specified time interval. Only meaningful in synchronous systems.Medium
Byzantine failureProcess sends arbitrary, incorrect, or malicious messages. The worst case — the faulty process may actively try to subvert the system.High
Network partitionThe network splits into two or more groups that cannot communicate with each other.High
Note

Most real-world distributed systems are designed to tolerate crash failures only. Tolerating Byzantine failures requires at least 3f + 1 nodes to survive f Byzantine faults — this is expensive and usually reserved for safety-critical or adversarial environments (blockchain, aircraft).

1.6 Scalability Dimensions

  • Size scalability — system can accommodate more users and resources.
  • Geographical scalability — system works across wider areas without performance degradation.
  • Administrative scalability — system remains manageable as it spans more organisations.

Scalability is limited by centralised components (single points of bottleneck), centralised data (single database serving all requests), and centralised algorithms (global knowledge required for decisions).

Networking & Communication

2.1 Inter-Process Communication (IPC)

Processes in a distributed system must communicate. The foundational abstraction is message passing — processes exchange discrete messages rather than sharing memory.

All higher-level communication abstractions (RPC, REST, sockets) are built on top of message passing primitives:

  • send(destination, message) — transmit a message to a destination process.
  • receive(source, &message) — wait to receive a message (blocking) or check if one is available (non-blocking).

2.2 Sockets and TCP/UDP

A socket is an abstraction over the network stack that provides an endpoint for communication. It is identified by (IP address, port number, protocol).

ReliabilityGuaranteed delivery, no duplicatesBest-effort, may lose/reorder/duplicate

2.3 Remote Procedure Call (RPC)

Definition

RPC allows a process to call a procedure on a remote machine as if it were a local procedure call, hiding the underlying message passing. Invented by Birrell and Nelson (1984).

CLIENT                              SERVER
──────                              ──────
1. Call local stub                  5. Server stub unpacks args
   result = add(3, 4)              6. Execute real function: add(3,4)
                                   7. Pack return value
2. Stub packs args (marshal)
3. Send message ──────────────────▶ 4. Server stub receives
8. Stub unpacks result (unmarshal)
9. Return 7 to caller ◀─────────── 8. Send result message

The components of an RPC system:

  • Client stub — marshals arguments, sends request, waits for reply, unmarshals result. Looks like the real function to the caller.
  • Server stub (skeleton) — receives request, unmarshals arguments, calls the real procedure, marshals result, sends reply.
  • Marshalling / Serialisation — converting in-memory data structures into a byte sequence suitable for transmission.
  • Interface Definition Language (IDL) — language-neutral description of the procedure signatures (e.g., Protocol Buffers for gRPC, WSDL for SOAP).

RPC Failure Semantics

SemanticGuaranteeHow achieved
MaybeProcedure executes 0 or 1 timesNo retries; simplest implementation
At-least-onceProcedure executes ≥1 timesClient retries on timeout; requires idempotent operations
At-most-onceProcedure executes 0 or 1 timesServer detects duplicate requests (via request ID) and returns cached result
Exactly-onceProcedure executes exactly 1 timeRequires at-most-once + persistent state + transactions; very difficult
Exam Tip

RPC failure semantics are heavily examined. Know that at-least-once is safe only for idempotent operations (operations that produce the same result regardless of how many times they're executed — e.g., setBalance(100) vs addToBalance(10)). Non-idempotent operations need at-most-once.

2.5 Message-Oriented Middleware (MOM)

Unlike RPC (synchronous — sender blocks), message-oriented middleware provides asynchronous communication via message queues or publish-subscribe systems.

ModelDescriptionExamples
Message QueuePoint-to-point. Producer sends to queue; one consumer reads. Guarantees at-least-once delivery.RabbitMQ, Amazon SQS, IBM MQ

MOM properties: temporal decoupling (sender and receiver don't need to be online simultaneously), spatial decoupling (sender doesn't know receiver identities), buffering (absorbs bursts of messages).

Time, Clocks & Global State

3.1 The Problem with Clocks

Physical clocks on different machines are never perfectly synchronised and drift continuously. This makes it impossible to determine a global ordering of events purely from timestamps. Two events on different machines with the same timestamp may have happened in either order — or simultaneously.

Clock synchronisation algorithms (NTP, PTP) can reduce drift to milliseconds or microseconds, but never to zero. This means we can never know which of two events with close timestamps happened first.

3.2 Logical Clocks — Lamport Timestamps

Leslie Lamport (1978) proposed logical clocks — scalar counters that capture causal ordering without requiring physical clock synchronisation.

Happened-Before Relation (→)

The happened-before relation captures causality. Event a → b (a happened before b) if: (1) a and b are in the same process and a occurred before b; (2) a is the sending of a message and b is the receipt of that same message; or (3) there exists an event c such that a → c and c → b (transitivity).

Lamport Clock Rules

  • Each process maintains a counter C, initialised to 0.
  • Internal event: Increment C by 1.
  • Send event: Increment C by 1; attach current C to message.
  • Receive event: Set C = max(C, C_message) + 1.

Property: If a → b, then L(a) < L(b). But the converse is not true — L(a) < L(b) does NOT imply a → b. Lamport clocks cannot detect concurrency.

3.3 Vector Clocks

Vector clocks (Fidge/Mattern, 1988) solve the limitation of Lamport clocks by tracking causality per-process.

Vector Clock

Each process P_i maintains a vector V_i[1..N] where N is the number of processes. V_i[j] is P_i's knowledge of how many events have occurred at P_j.

Vector Clock Rules

  • Internal event at P_i: Increment V_i[i] by 1.
  • Send at P_i: Increment V_i[i] by 1; attach V_i to message.
  • Receive at P_j from P_i with timestamp V_m: Set V_j[k] = max(V_j[k], V_m[k]) for all k; then increment V_j[j] by 1.

Comparing Vector Clocks

  • V_a = V_b iff V_a[k] = V_b[k] for all k.
  • V_a ≤ V_b iff V_a[k] ≤ V_b[k] for all k.
  • a → b iff V_a < V_b (V_a ≤ V_b and V_a ≠ V_b).
  • a ∥ b (concurrent) iff V_a ≰ V_b and V_b ≰ V_a.
Process P1        Process P2        Process P3
──────────        ──────────        ──────────
[1,0,0] a
                  [0,1,0] b
[2,0,0] c ──────▶ [2,2,0] d
                  [2,3,0] e ──────▶ [2,3,1] f

a → c  (same process, c later)
b → d  (message received)
c → d  (c sent, d received that message)
b ∥ a  (concurrent — neither happened-before the other)

3.4 Global State & Consistent Cuts

A global state is a snapshot of the states of all processes and all messages in transit at some instant. But there is no global clock to take such a snapshot simultaneously.

Consistent Cut

A cut is a set of events — one prefix of each process's history. A cut is consistent if for every event e in the cut, all events that happened-before e are also in the cut. Equivalently: if a receive event is in the cut, the corresponding send event must also be in the cut.

3.5 The Chandy-Lamport Snapshot Algorithm

This algorithm records a consistent global snapshot of a running distributed system without stopping it. Used for checkpoint/restart, deadlock detection, and distributed garbage collection.

Assumptions: channels are FIFO; processes don't crash; the initiator is known.

Algorithm:

  1. Initiator P_i records its own state; sends a marker message on all outgoing channels.
  2. When process P_j receives a marker on channel C for the first time: record P_j's state; record the state of channel C as empty; send a marker on all outgoing channels.
  3. When P_j subsequently receives a marker on another channel C': record the state of C' as all messages received on C' since P_j recorded its state.
  4. Terminate when all processes have received a marker on all incoming channels.
Exam Tip

The most common exam question on vector clocks is to trace through a given message-passing diagram and compute the final vector clock for each event, then determine which pairs of events are concurrent. Remember: concurrent means neither happened-before the other — the clocks are incomparable in both directions.

Mutual Exclusion

4.1 The Problem

In a distributed system with no shared memory, ensuring that only one process at a time accesses a critical section (shared resource) requires explicit coordination via message passing. There is no mutex or semaphore system call.

Requirements for a correct distributed mutual exclusion algorithm:

  • Safety (ME): At most one process is in the critical section at any time.
  • Liveness: Every request to enter the critical section eventually succeeds (no deadlock, no starvation).
  • Ordering (desirable): Requests are granted in happened-before order.

4.2 Centralised Algorithm

One process acts as a coordinator. Others request permission, wait for a grant, and release when done.

P1 ──REQUEST──▶  Coordinator  ──GRANT──▶ P1 (enters CS)
P2 ──REQUEST──▶  Coordinator  (queues P2)
P1 ──RELEASE──▶  Coordinator  ──GRANT──▶ P2 (enters CS)
Centralised
Messages per CS entry3 (request, grant, release)
Delay before entry2 message delays
ProblemsSingle point of failure; coordinator bottleneck

4.3 Ricart-Agrawala Algorithm (Distributed)

A fully distributed algorithm based on Lamport timestamps. No coordinator needed.

To enter CS, process P_i:

  1. Increments its Lamport clock; records timestamp T.
  2. Sends REQUEST(T, i) to all other N-1 processes.
  3. Waits until it has received REPLY from all N-1 processes.

When P_j receives REQUEST(T, i) from P_i:

  • If P_j is not interested in CS and not in CS: immediately send REPLY.
  • If P_j is in CS: queue the request; send REPLY after exiting.
  • If P_j has requested CS: compare timestamps. If P_j's timestamp is smaller (higher priority), queue P_i's request. Otherwise send REPLY to P_i immediately.
Ricart-Agrawala
Messages per CS entry2(N-1) (N-1 requests + N-1 replies)
Delay before entry2 message delays
ProblemsAll nodes must be operational; O(N) messages

4.4 Token Ring Algorithm

Processes are arranged in a logical ring. A single token circulates continuously. A process may only enter the CS when it holds the token.

Token Ring
Messages per CS entry0 to N-1 (depends on token position)
Delay before entry0 to N-1 message delays
ProblemsToken loss must be detected and regenerated; token circulates even when nobody wants CS (bandwidth wasted)

4.5 Maekawa's Algorithm (Quorum-Based)

Instead of seeking permission from all N processes (Ricart-Agrawala) or a single coordinator, Maekawa requires permission only from a quorum — a subset of processes.

Each process P_i has a voting set V_i ⊆ {P_1, ..., P_N} such that:

  • P_i ∈ V_i (each process is in its own voting set).
  • For any i, j: V_i ∩ V_j ≠ ∅ (any two voting sets share at least one member).
  • |V_i| = K ≈ √N for all i.

Messages per entry: 3√N (lower than Ricart-Agrawala for large N). Handles failures better. Risk of deadlock — requires additional deadlock handling.

Leader Election

5.1 Why Election?

Many distributed algorithms require a coordinator — a designated process that makes decisions for the group (e.g., distributed mutex coordinator, primary in primary-backup replication, master in MapReduce). When the coordinator crashes, the system must elect a new one automatically. This is the leader election problem.

Election Problem

Given N processes each with a unique identifier, elect exactly one process as the leader such that: (1) every non-faulty process eventually agrees on the same leader; (2) the elected process knows it is the leader.

5.2 The Bully Algorithm

Assumes synchronous system (timeouts are reliable). The process with the highest ID wins.

When P_i detects coordinator failure:

  1. P_i sends ELECTION to all processes with higher IDs.
  2. If P_i receives no OK within timeout: P_i declares itself leader, sends COORDINATOR to all lower-ID processes.
  3. If P_i receives OK from higher-ID process: P_i waits for COORDINATOR message.
  4. Any process receiving ELECTION replies OK and starts its own election (bullies the sender).

Complexity: O(N²) messages in the worst case. Problem: The "bully" behaviour causes a temporarily crashed (then recovered) high-ID process to constantly re-elect itself.

5.3 The Ring Election Algorithm

Processes are arranged in a logical ring. Each process knows its successor.

  1. P_i detects coordinator failure; sends ELECTION(i) to successor.
  2. Each process receiving ELECTION(id): forwards ELECTION(max(id, own_id)) to successor.
  3. When P_i receives ELECTION(i) (its own ID came back around): P_i is the leader; sends COORDINATOR(i) around the ring to inform all.

Complexity: 2N - 1 messages in the best case, 3N - 1 in the worst. Lower overhead than Bully.

5.4 Raft Leader Election

Raft (Ongaro & Ousterhout, 2014) uses a randomised timeout approach to leader election, avoiding the need for complex tie-breaking.

Each server is in one of three states: Follower, Candidate, Leader.

               times out,
               starts election
                    │
  Follower ─────────┼──────▶ Candidate ──receives majority votes──▶ Leader
     ▲              │             │                                     │
     │         (initially)        │ discovers current leader            │
     │                            ▼ or new term                         │
     └──────────────────────── Follower ◀───────────────────────────────┘
                                           discovers server with higher term

Each Raft node has an election timeout (randomly chosen between 150–300ms). If a follower receives no heartbeat from a leader within this timeout, it becomes a candidate and starts an election for a new term. It votes for itself and requests votes from all peers. A majority vote wins. Randomisation ensures candidates rarely start elections simultaneously.

Exam Tip

Election algorithms are examined both conceptually (describe the algorithm) and analytically (trace through an execution, count messages). Know the message complexity of each: Bully — O(N²) worst case; Ring — O(N); Raft — randomised, typically O(N) per election. Also know why Bully is called "bully" — higher-ID processes dominate lower-ID ones aggressively.

Consensus

6.1 The Consensus Problem

Consensus

A set of processes must agree on a single value. Each process proposes a value; after executing the protocol, all non-faulty processes must decide on the same value (that was proposed by at least one process).

Consensus must satisfy three properties:

  • Agreement: All non-faulty processes decide on the same value.
  • Validity: The decided value was proposed by some process.
  • Termination: All non-faulty processes eventually decide.

6.2 The FLP Impossibility Result

Fischer, Lynch, Paterson (1985)

FLP Impossibility: In a fully asynchronous distributed system, there is no deterministic algorithm that solves consensus in the presence of even a single crash failure.

This is one of the most fundamental results in distributed computing. The proof shows that for any proposed protocol, there always exists an execution where the protocol never terminates. The intuition: in an asynchronous system, a crashed process is indistinguishable from a slow one. A protocol that terminates despite a crash may terminate incorrectly if the process was merely slow.

Implications: To achieve consensus, we must relax at least one assumption:

  • Assume partial synchrony (the real world) — the system is asynchronous most of the time, but occasionally periods of synchrony allow progress.
  • Use randomisation to break symmetry (achieve consensus with high probability).
  • Sacrifice safety — allow processes to occasionally decide differently (not suitable for most systems).

6.3 Paxos

Paxos (Lamport, 1989/1998) is the foundational consensus algorithm. It operates in two phases and tolerates up to f crash failures with 2f+1 nodes.

Paxos roles: Proposer (proposes values), Acceptor (votes on proposals), Learner (learns the decided value). One node may play multiple roles.

Phase 1 — Prepare / Promise

Paxos Phase 1
Proposer selects proposal number n (must be unique and higher than any seen before)

Proposer → ALL Acceptors:   PREPARE(n)

Each Acceptor receiving PREPARE(n):
  if n > highest_promised:
    set highest_promised = n
    send PROMISE(n, accepted_n, accepted_v) to Proposer
        // accepted_n: highest proposal number this acceptor has accepted
        // accepted_v: value accepted with that proposal (null if none)
  else:
    ignore (or send NACK)

Phase 2 — Accept / Accepted

Paxos Phase 2
If Proposer receives PROMISE from majority of Acceptors:
  if any PROMISE contained an accepted value:
    v = accepted_v with the highest accepted_n among received promises
  else:
    v = Proposer's own proposed value

Proposer → Majority of Acceptors:   ACCEPT(n, v)

Each Acceptor receiving ACCEPT(n, v):
  if n >= highest_promised:
    accept the proposal
    send ACCEPTED(n, v) to all Learners
  else:
    ignore

Learners: upon receiving ACCEPTED(n, v) from majority → value v is decided

6.4 Raft Consensus

Raft (Ongaro & Ousterhout, 2014) was designed to be more understandable than Paxos while providing equivalent guarantees. It decomposes consensus into three sub-problems:

  • Leader election — select one server as leader (Week 5).
  • Log replication — leader accepts client requests, appends to its log, replicates to followers.
  • Safety — ensure only servers with up-to-date logs can become leaders.

Log Replication in Raft

Raft Log Replication
1. Client sends command to Leader
2. Leader appends entry to its log: [term=T, index=I, command=cmd]
3. Leader sends AppendEntries RPC to all followers:
   { term, leaderId, prevLogIndex, prevLogTerm, entries[], leaderCommit }
4. Followers check consistency:
   - if prevLogIndex/prevLogTerm matches local log: append entries, reply OK
   - if mismatch: reply FAIL (leader then backs up and retries)
5. Once majority ACK: Leader commits entry (applies to state machine)
6. Leader notifies followers of commit via next AppendEntries
7. Followers apply committed entries to their state machines

6.5 Byzantine Fault Tolerant (BFT) Consensus

Standard Paxos and Raft only tolerate crash failures. Byzantine fault tolerance is required when processes may send arbitrary or malicious messages.

PBFT (Practical Byzantine Fault Tolerance) — Castro & Liskov, 1999. Requires 3f+1 replicas to tolerate f Byzantine faults. Three phases: pre-prepare, prepare, commit. O(N²) message complexity per consensus round.

Replication & Consistency

7.1 Why Replicate?

  • Fault tolerance: If one replica fails, others continue serving requests.
  • Performance: Serve reads from the nearest replica; distribute load.
  • Availability: System remains operational despite partial failures.

The fundamental challenge: keeping replicas consistent. Every write must be propagated to all replicas. The tension between consistency, availability, and performance is formalised by the CAP theorem.

7.2 The CAP Theorem

Brewer (2000), Proved by Gilbert & Lynch (2002)

It is impossible for a distributed system to simultaneously provide all three of: Consistency (every read receives the most recent write), Availability (every request receives a response), and Partition tolerance (the system continues operating despite network partitions).

              Consistency
              /          \
             /            \
            /              \
           /   CA           \
          /  (no partition)  \
         /___________________\
        /                     \
       /   CP        AP        \
      /  (forfeit     (forfeit  \
     /   avail.)     consist.)  \
    /___________________________\
 Availability      Partition Tolerance

Since network partitions are unavoidable in any real distributed system, the practical trade-off is CP vs AP:

  • CP systems — refuse to serve stale data during a partition (return error or block). Example: HBase, Zookeeper, traditional RDBMS.
  • AP systems — continue serving requests during a partition but may return stale data. Example: Cassandra, DynamoDB, CouchDB.

7.3 Consistency Models

Consistency models define the contract between the storage system and the application — what guarantees does the application have about the order in which it sees writes?

ModelGuaranteeStrength
LinearisabilityOperations appear to execute atomically at some point between their invocation and completion. The strongest model. Equivalent to having a single server.Strongest
Sequential consistencyAll processes see all writes in the same order; that order is consistent with each process's own order. But operations don't need to appear to happen "at their real time".Strong
Causal consistencyCausally related writes are seen in order by all processes. Concurrent writes may be seen in different orders by different processes.Medium
Eventual consistencyIf no new writes are made to an item, all replicas will eventually converge to the same value. No guarantee on when or what intermediate values are seen.Weak
Read-your-writesA client always sees its own previous writes. A session-level guarantee.Session
Monotonic readsA client never sees an older version of a value after seeing a newer one.Session

7.4 Replication Protocols

Primary-Backup (Primary-Secondary)

One replica is designated primary; it handles all writes and propagates them to backup replicas.

  • Synchronous replication: Primary waits for backups to acknowledge write before responding to client. Strong consistency; higher write latency.
  • Asynchronous replication: Primary responds to client immediately; propagates to backups later. Low write latency; risk of data loss on primary crash.

Multi-Master (Multi-Primary)

Multiple replicas accept writes simultaneously. Writes must be merged and conflicts resolved. Common approaches: last-write-wins (LWW), vector clock versioning, CRDTs.

7.5 CRDT — Conflict-Free Replicated Data Types

CRDTs are data structures designed so that concurrent operations always merge correctly without coordination, by construction. Any two CRDTs representing the same set of operations will converge to the same state.

  • G-Counter (Grow-only Counter) — each replica maintains its own count; total = sum of all replicas.
  • LWW-Register — last-write-wins based on timestamp.
  • G-Set (Grow-only Set) — elements can only be added, never removed; merge = union.
  • OR-Set (Observed-Remove Set) — allows both add and remove; uses unique tags to disambiguate concurrent add/remove.

Fault Tolerance

8.1 Dependability Properties

Fault tolerance aims to achieve dependability — the trustworthiness of a system. Dependability encompasses:

PropertyDefinition
AvailabilitySystem is operational and accessible at a given instant. Measured as uptime percentage (e.g., "five nines" = 99.999% = ~5 minutes downtime/year).
ReliabilitySystem runs continuously without failure for a given time interval. A system can be highly available but unreliable (lots of brief failures that restart quickly).
SafetyWhen system fails, it fails safely — no catastrophic consequences.
MaintainabilityHow easily a failed system can be repaired and restored.

8.2 Failure Detection

In an asynchronous system, a failed process is indistinguishable from a very slow one. Failure detectors provide the abstraction of deciding (possibly incorrectly) whether a process has failed.

Chandra & Toueg (1996) characterise failure detectors by two properties:

  • Completeness: Every crashed process is eventually suspected by every correct process.
  • Accuracy: No correct process is ever (or eventually is not) suspected.

The heartbeat mechanism: each process sends periodic "I'm alive" messages. If a heartbeat is not received within a timeout, the process is suspected of failure.

8.3 Process Replication for Fault Tolerance

k-Fault Tolerance

A system is k-fault tolerant if it continues to function correctly even when k components fail. To tolerate k crash failures, a system needs at least k+1 replicas. To tolerate k Byzantine failures, it needs at least 2k+1 (for masking by majority vote) or 3k+1 (for full BFT consensus).

Active Replication (State Machine Replication)

All replicas are active; they all process every request. Client broadcasts request to all replicas; all replicas execute and reply. Requires total order broadcast (atomic multicast) to ensure all replicas see requests in the same order. Used in Paxos/Raft-based systems.

Passive Replication (Primary-Backup)

Only the primary processes requests. Primary checkpoints state to backups. On primary failure, a backup takes over. Simpler than active replication but has higher recovery latency.

8.4 Reliable Group Communication

Reliable multicast is needed for replication protocols. It builds on basic network multicast to provide stronger guarantees:

ProtocolGuarantee
Reliable multicastIf any correct process delivers message m, all correct processes deliver m
FIFO multicastMessages from the same sender are delivered in send order
Causal multicastIf send(m1) → send(m2), then deliver(m1) before deliver(m2) at all processes
Total order multicast (atomic broadcast)All processes deliver all messages in exactly the same order
Key Insight

Total order multicast is equivalent to consensus. Solving one solves the other. If you can totally order messages (atomic broadcast), you can implement consensus by proposing your value as a message and using the first message delivered as the decided value.

8.5 Recovery

When a failed process restarts, it must recover its state. Two main approaches:

  • Checkpoint and rollback: Periodically save process state to stable storage. On restart, roll back to the most recent consistent checkpoint. Must coordinate checkpoints across processes to avoid the domino effect (rolling back triggers more rollbacks due to message dependencies).
  • Message logging: Log all received messages. On restart, replay messages from the log to restore state without rolling back to an old checkpoint. Sender-based vs receiver-based logging.

Distributed Transactions

9.1 ACID Properties

ACID

ACID is the set of properties that guarantee correctness of database transactions, even in the presence of errors, power failures, and concurrent accesses.

PropertyDefinition
AtomicityA transaction executes completely or not at all. No partial updates. "All or nothing."
ConsistencyA transaction brings the database from one valid state to another, preserving all defined rules (constraints, cascades, triggers).
IsolationConcurrent transactions execute as if they were serial. Intermediate states are not visible to other transactions.
DurabilityOnce a transaction commits, it remains committed even in the event of system failure. Written to non-volatile storage.

9.2 Distributed Transactions

A distributed transaction spans multiple nodes, each managing a different part of the data. All nodes must either commit or abort together — otherwise the system is left in an inconsistent state.

9.3 Two-Phase Commit (2PC)

The standard protocol for achieving atomic commit across multiple participants.

Phase 1 — Voting
─────────────────
Coordinator ──PREPARE──▶ Participant A
Coordinator ──PREPARE──▶ Participant B
Coordinator ──PREPARE──▶ Participant C

Participant A ──VOTE-COMMIT──▶ Coordinator  (logged to stable storage)
Participant B ──VOTE-COMMIT──▶ Coordinator
Participant C ──VOTE-ABORT──▶  Coordinator

Phase 2 — Decision
────────────────────
If all votes = COMMIT:
  Coordinator ──GLOBAL-COMMIT──▶ all participants
Else (any ABORT or timeout):
  Coordinator ──GLOBAL-ABORT──▶ all participants

Participants apply decision and release locks.

2PC Failure Analysis

Failure scenarioOutcome
Participant crashes before votingCoordinator times out; aborts transaction
Participant crashes after voting COMMITParticipant must block until coordinator decision is received on recovery
Coordinator crashes before sending decisionBlocking problem: Participants that voted COMMIT are blocked — they cannot decide alone. This is 2PC's fundamental limitation.
Network partitionParticipants on minority side block indefinitely
2PC Blocking Problem

2PC is a blocking protocol. If the coordinator fails after participants have voted COMMIT but before sending the global decision, participants holding locks cannot release them until the coordinator recovers. This is why 2PC is unacceptable for high-availability systems — a single coordinator failure causes all participants to block indefinitely.

9.4 Three-Phase Commit (3PC)

3PC adds a pre-commit phase between voting and committing, enabling participants to make a commit decision without the coordinator in some failure scenarios.

3PC is non-blocking in the absence of network partitions but does not tolerate partitions — it can lead to a split-brain where some participants commit and others abort during a partition. Therefore, 3PC is rarely used in practice.

9.5 Concurrency Control

Multiple transactions executing concurrently must be isolated. The standard isolation levels (ANSI SQL):

Isolation LevelDirty ReadNon-repeatable ReadPhantom Read
Read UncommittedPossiblePossiblePossible
Read CommittedPreventedPossiblePossible
Repeatable ReadPreventedPreventedPossible
SerialisablePreventedPreventedPrevented

Two-Phase Locking (2PL)

The classic approach to achieving serialisability. A transaction has two phases: a growing phase (acquires locks, never releases) and a shrinking phase (releases locks, never acquires). Guarantees serialisability. Risk: deadlock.

Optimistic Concurrency Control (OCC)

Transactions execute without locking; at commit time, check for conflicts with other committed transactions. If a conflict is found, abort and retry. Low overhead for low-contention workloads; expensive for high-contention.

Multiversion Concurrency Control (MVCC)

Each write creates a new version of a data item. Readers access the version current at their transaction's start time. Reads never block writes; writes never block reads. Used in PostgreSQL, CockroachDB, Google Spanner.

Peer-to-Peer Systems & Distributed Hash Tables

10.1 P2P Architecture

In a peer-to-peer system, all nodes (peers) are functionally equivalent — they act as both clients and servers. There is no permanent centralised coordinator. Resources (files, computation) are distributed across peers.

Generations of P2P systems:

  • Centralised index (Napster): Central server indexes which peer has which file. Peers exchange files directly. Single point of failure and legal liability.
  • Flooding (Gnutella): No central index. Queries are flooded to all peers within a TTL hop limit. Very inefficient — O(N) messages per query.
  • Structured P2P (DHTs): Routing tables organised by a distributed hash function. Guaranteed lookup in O(log N) messages.

10.2 Distributed Hash Tables (DHTs)

DHT

A Distributed Hash Table is a decentralised distributed system that provides a lookup service similar to a hash table: key → value pairs are distributed across nodes, and any node can efficiently retrieve the value associated with any key.

Key properties of a DHT:

  • Keys and node IDs live in the same identifier space (typically [0, 2^m - 1]).
  • Each node is responsible for storing key-value pairs for the keys closest to its own ID.
  • Lookup routes a query towards the node responsible for a key in O(log N) hops.
  • The system self-heals when nodes join or leave (churn).

10.3 Chord

Chord (Stoica et al., 2001) is the canonical DHT. Nodes and keys are assigned m-bit IDs; arranged in a logical ring modulo 2^m.

Successor: Each key k is stored at the node with the smallest ID ≥ k. This node is successor(k).

Finger table: Each node maintains a routing table of m entries:

  • Entry i points to: successor(n + 2^(i-1)) mod 2^m
  • Finger[1] = immediate successor; Finger[m] = node roughly halfway around the ring.
  • Lookup uses the finger table to jump halfway each time → O(log N) hops.
Chord ring (m=4, so IDs 0–15):

          0
       15   1
     14       2
    13         3    Node IDs: {1, 4, 7, 11}
    12         4 ●  Key 6 → stored at node 7 (successor of 6)
    11 ●       5
     10        6
       9    7 ●
          8

Node 1's finger table (m=4):
  i=1: successor(1+1=2)   → node 4
  i=2: successor(1+2=3)   → node 4
  i=3: successor(1+4=5)   → node 7
  i=4: successor(1+8=9)   → node 11

Lookup of key 6 from node 1:
  Finger closest to 6 without exceeding it → node 4
  Node 4 forwards to → node 7 (successor of 6)
  Node 7 returns value. (2 hops)

10.4 Kademlia

Kademlia (Maymounkov & Mazières, 2002) uses XOR distance metric instead of ring topology. Distance between two IDs = XOR of their bit representations. XOR is symmetric and obeys triangle inequality.

Each node maintains k-buckets — routing table entries grouped by distance. Lookup converges in O(log N) steps. Used in BitTorrent DHT, Ethereum, IPFS. More resilient to churn than Chord.

10.5 Consistent Hashing

Consistent hashing (used in Dynamo, Redis Cluster, load balancers) solves the problem of adding/removing nodes without remapping all keys.

Both nodes and keys are hashed onto a ring [0, 2^m). Each key is assigned to the first node clockwise from its position. When a node is added/removed, only keys in one segment of the ring are remapped (on average N/K keys where K is the total number of keys).

Virtual nodes: Each physical node is represented by multiple virtual nodes on the ring, improving load balancing.

Distributed Storage Systems

11.1 Design Principles of Distributed Storage

Large-scale distributed storage systems are designed around a core set of principles that differ markedly from single-node databases:

  • Design for failure: Hardware failures are the norm, not the exception. Data and computation must be replicated.
  • Move computation to data: Network bandwidth is the bottleneck; processing data where it lives (not copying to a central processor) is more efficient.
  • Commodity hardware: Use cheap, unreliable nodes rather than expensive, reliable ones. Reliability comes from software.
  • Simple consistency model: Relaxed consistency (eventual consistency) enables much higher performance and availability.

11.2 Google File System (GFS)

GFS (Ghemawat, Gobioff, Leung 2003) is the foundational paper for distributed file systems at internet scale. Key design decisions:

  • Large files, large chunks: Chunk size = 64MB (vs 4KB in Unix). Reduces metadata overhead; good for batch sequential reads.
  • Single master: Stores metadata (namespace, chunk locations, access control). All clients communicate with master for metadata, then directly with chunkservers for data.
  • Replication factor 3: Each chunk is replicated on 3 chunkservers. On failure, master re-replicates automatically.
  • No caching: Files are too large to cache usefully. Sequential access patterns make caching ineffective.
  • Append-only semantics: Optimised for record appends (log files, crawls). Random writes are discouraged.
  • Relaxed consistency: GFS allows "record append" without strict ordering guarantees — different replicas may have gaps or duplicates.

11.3 MapReduce

MapReduce (Dean & Ghemawat, 2004) is a programming model and execution framework for processing large datasets in parallel across a cluster.

Input data (split across HDFS blocks)

Word Count example:

MapReduce — Word Count
# Map function: for each word in the input line, emit (word, 1)
def map(line):
    for word in line.split():
        emit(word, 1)

# Reduce function: sum all counts for a given word
def reduce(word, counts):
    emit(word, sum(counts))

# The MapReduce framework handles:
# - Splitting input across mappers
# - Distributing map tasks across the cluster
# - Shuffling and sorting intermediate results by key
# - Distributing reduce tasks
# - Handling failures (re-executing failed tasks)

11.4 Amazon Dynamo

Dynamo (DeCandia et al., 2007) is Amazon's highly available key-value store. It prioritises availability over strong consistency — an AP system.

Key design choices:

ProblemDynamo's Solution
PartitioningConsistent hashing with virtual nodes
ReplicationPreference list — replicate to N successor nodes on the ring
ConsistencyEventual consistency; vector clocks for versioning; quorum reads/writes
Failure detectionGossip-based heartbeating
Conflict resolutionApplication-level (shopping cart merge) or last-write-wins
Sloppy quorumHinted handoff — if preferred replica is down, temporarily store on another node with a "hint"

Dynamo Quorum System

With N replicas, write quorum W, read quorum R:

  • Write: Wait for W replicas to acknowledge before returning success.
  • Read: Query R replicas and return the most recent version.
  • Strong consistency condition: R + W > N. Ensures read and write sets overlap.
  • Typical Dynamo settings: N=3, W=2, R=2 → R+W=4>3 (strong consistency possible but slower). Or W=1, R=1 → eventual consistency, maximum availability.

11.5 Google Spanner

Spanner (Corbett et al., 2012) is Google's globally distributed SQL database. It achieves external consistency (equivalent to linearisability) at global scale using TrueTime — a globally synchronised clock API that bounds clock uncertainty to within ~7ms using GPS and atomic clocks.

Spanner uses this bounded uncertainty to implement commit-wait: a transaction's commit timestamp must be past the end of the uncertainty interval before the transaction is visible. This ensures transactions are globally ordered by their commit timestamps.

Security & Course Summary

12.1 Security in Distributed Systems

Distributed systems face a substantially larger attack surface than centralised systems. Every communication channel is a potential attack vector; every node is a potential point of compromise.

Security goals in distributed systems:

  • Confidentiality: Information is disclosed only to authorised principals.
  • Integrity: Information is modified only by authorised principals.
  • Availability: Resources are accessible to authorised principals when needed.
  • Authentication: The identity of principals is correctly verified.
  • Non-repudiation: Actions cannot be denied after the fact.

12.2 Cryptographic Foundations

Symmetric Encryption

Same key for encryption and decryption. Fast (hardware acceleration). Problem: key distribution — how do two parties agree on a shared key over an insecure channel? Examples: AES-256, ChaCha20.

Asymmetric (Public Key) Encryption

Two mathematically related keys: public key (distribute freely) and private key (keep secret). Data encrypted with public key can only be decrypted with the private key. Solves key distribution. Slow — used for key exchange, not bulk data. Examples: RSA-2048, ECDH.

Diffie-Hellman Key Exchange

Allows two parties to agree on a shared secret over an insecure channel, without ever transmitting the secret itself. The basis for TLS forward secrecy.

Diffie-Hellman (simplified)
Public parameters: prime p, generator g (known to everyone)

Alice: chooses secret a, computes A = g^a mod p, sends A to Bob
Bob:   chooses secret b, computes B = g^b mod p, sends B to Alice

Alice computes shared secret: S = B^a mod p = g^(ab) mod p
Bob   computes shared secret: S = A^b mod p = g^(ab) mod p

Eve sees: p, g, A, B — computing a or b from these requires solving
the discrete logarithm problem (computationally infeasible for large p)

Digital Signatures

Alice signs a message with her private key. Anyone with Alice's public key can verify the signature. Provides authentication and non-repudiation. Used in TLS certificates, code signing, distributed consensus protocols.

12.3 Authentication in Distributed Systems

Kerberos

Kerberos is the dominant authentication protocol for enterprise distributed systems (used in Active Directory, which connects this unit directly back to SOC).

Client (C)          Key Distribution Center (KDC)           Service (S)
                    ┌──────────────┬───────────────────┐
                    │ Auth Server  │  Ticket Granting  │
                    │    (AS)      │   Server (TGS)    │
                    └──────────────┴───────────────────┘

① C ──AS_REQ(C, TGS)──────────────────────────────▶ AS
② C ◀──AS_REP(TGT encrypted with TGS key, session key K_c,tgs)── AS

③ C ──TGS_REQ(TGT, authenticator, service S)──────▶ TGS
④ C ◀──TGS_REP(service ticket encrypted with K_s, K_c,s)── TGS

⑤ C ──AP_REQ(service ticket, authenticator)──────▶ S
⑥ C ◀──AP_REP(authenticator+1)──────────────────── S

TGT = Ticket Granting Ticket  (reused for step ③ — no password re-entry)
K_s = Service's secret key    (shared between KDC and service)

Public Key Infrastructure (PKI)

PKI binds public keys to identities through a hierarchy of trusted Certificate Authorities (CAs). A certificate is a digitally signed statement: "CA asserts that this public key belongs to this entity."

TLS (Transport Layer Security) uses PKI to authenticate servers and establish encrypted sessions for all HTTPS traffic.

12.5 The Eight Fallacies of Distributed Computing

Peter Deutsch (Sun Microsystems, 1994) — assumptions that developers incorrectly make when building distributed systems for the first time:

  1. The network is reliable.
  2. Latency is zero.
  3. Bandwidth is infinite.
  4. The network is secure.
  5. Topology doesn't change.
  6. There is one administrator.
  7. Transport cost is zero.
  8. The network is homogeneous.
Final Exam Tip

The most commonly examined topics across this unit are: vector clock computation (trace an execution, identify concurrent events); CAP theorem application (classify a system as CP or AP, justify based on its behaviour during partitions); Paxos phases (trace a Paxos execution, identify what happens when the proposer or an acceptor crashes mid-protocol); 2PC failure modes (identify which failure causes blocking and why); and consistency model comparison (give an execution that satisfies eventual consistency but not causal consistency). Master these five areas and you cover the majority of examination weight.
